目标
修复 agent 在“修改购物车数量”流程中：当用户触发修改数量意图时，系统应立即让 agent 解析参数并继续执行候选项选择或后端调用，而不是弹出只有一个 `cancel` 的“助手提示”并结束会话。解决 item_id 为 null 导致前端 alert 且未触发“如果后端提供 candidate_items，则提示用户选择”的问题。

问题摘要
- 场景：用户想修改购物车中某个商品数量（增/删操作正常，改数量失败）。
- 现象：agent 弹出一个只含 `cancel` 的助手提示，点击 `cancel` 后对话结束；前端收到 `item_id = null` 导致直接 `alert` 并停止流程；后端提供 `candidate_items` 时没有触发“提示用户选择”的逻辑。 

目标结果
- agent 在接收到「修改数量」意图后：
  1) 立即解析用户意图和槽位（商品名称、规格、目标数量、上下文），
  2) 若 item_id 不确定但后端返回 candidate_items，前端/agent 应提示用户选择（多选/确认），而不是直接 alert 或结束对话，
  3) 若 item_id 明确，则正常发起修改请求并返回确认，
  4) 避免出现只含 cancel 的确认弹窗终止会话的情况。

3. 跟踪一个典型失败请求的控制流：从用户消息进入到 agent，哪里决定弹出只含 cancel 的提示，item_id 如何被设置为 null。查看是否有异常吞噬或 premature return。 

假设（先验猜测）
- intent 解析正常，但在构造“助手提示”用于确认时，生成的交互模板缺少“确认/继续”按钮（仅保留 cancel），导致前端只有取消途径；
- item_id 在解析槽位时未被正确填充（实体识别或上下文回溯失败）或被错误覆盖为 null；
- 当后端返回 candidate_items 时，agent 没有把它们转换为前端可用的选择菜单（或使用了错误的消息类型），因此“如果后端提供 candidate_items, prompt user to choose which cart item”这段逻辑没有执行或被短路。

修复策略（按影响与风险排序）
1) Fix confirmation UI generation: 确保在生成“确认修改数量”提示时包含明确的操作按钮（例如：确认/取消、选择候选项列表），不要只生成 cancel。修改消息模板或默认按钮集合，保证至少存在一个继续操作按钮。低风险、优先级高。
2) Ensure candidate_items flow: 如果后端返回 candidate_items 且 item_id 为 null，agent 应向前端发送一个选择式消息（quick replies 或选择 modal），并将会话保持为等待用户选择的状态，而非结束会话或弹出单一 cancel。中风险。
3) Fix slot-to-item mapping: 修复或加固槽位填充逻辑，确保从用户语句和上下文提取到的商品标识映射到 item_id；如果模糊则触发 candidate_items 分支。若是 NLU 模块问题，增加后备策略（模糊匹配、基于商品名检索）。中高风险。
4) Add defensive checks: 在前端收到 `item_id = null` 时不要直接 alert；改为触发 candidate_items 选择或向后端请求候选项。短期补丁优先级高以减少用户端报错。低风险前端改动。

新出现的问题与修复建议
问题：当用户意图是加入新书（add）且购物车中已存在其他书目时，前端会错误地弹出“选择 item_id”的候选框（或进入 update 流），而不是直接解析用户选定的书与版本并执行 add 操作。

根因分析（初步）：
- 之前的修复在 action_suggestion 中只要检测到 `item_id is None` 或 `missing_fields` 包含 `item_id` 就会为当前会话的 cart_payload 注入 `candidate_items`，并将 `should_act=False`。此逻辑没有区分 action 类型（add/update/remove），导致在 add 场景中也会注入候选项并阻断 add 流，前端接收到后把流程误判为需要选择已有 cart item（update/remove）。

修复策略（具体）：
1) 限制后端注入 candidate_items 的情形：仅在 action 为 `update` 或 `remove` 时，才在 `action_suggestion` 中添加 `candidate_items`、将 `missing_fields` 包含 `item_id` 并把 `should_act=False`。对于 `add` 操作，不应该附带 candidate_items（因为 add 针对的是 catalog references / sku，而非已有 cart items）。
2) 保证 `add` 路径优先处理 `sku_requests` / `sku_id` / reference_payload：如果 DeepSeek 给出 add 且包含 sku_requests 或 sku_id，应继续进入 add 的确认流程（不要被后面的 cart-based candidate_items 分支覆盖）。
3) 前端补强：当前前端已有对 add 的处理（基于 sku_requests / sku_id）；保持不变，但确保当后端错误地返回 candidate_items 时前端不会把 add 自动转为 update — 可选地在前端优先判断 suggestion.action === 'add' 并绕过 item_id 解析分支。
