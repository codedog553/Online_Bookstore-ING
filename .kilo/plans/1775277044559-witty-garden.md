# 书店测试数据库接入计划

## 目标

为当前项目引入“可公开获取的真实书籍图片”，优先解决前端与 demo 场景中的封面展示问题；在不重做整套数据库方案的前提下，把现有 `.txt` 伪图片替换成真实可展示图片，使列表、详情、轮播与后台图片管理看起来是完成态。

## 用户新增约束

用户已明确：当前阶段只需要真实图片，不需要一次性把整套开源书籍数据库全部接入；核心诉求是不再使用占位符图片，否则 demo 时会显得图片功能未完成。

进一步约束已明确：

- 不仅要“有真实图片”，还要尽量避免反复出现相同 demo 书名和 demo 图片
- 商品列表、详情页、后台商品管理里应尽量显示现实存在的书籍名、作者、出版社、简介与封面
- 图片下载机制要比当前更稳定，不能把 demo 成败建立在运行时临时联网之上
- 用户新增偏好：优先使用“非本地人工素材”，即优先采用开源数据库/公开 API 提供的真实书籍照片与书籍信息；本地缓存仅作为稳定性兜底，而不是主数据来源

## 现状结论

### 1. 当前项目对数据的真实要求

从代码与 README 可确认，项目不是单纯的“图书列表展示”，而是要求一套可驱动完整业务流程的数据：

- 商品基础信息：`title/title_en`、`author/author_en`、`publisher/publisher_en`、`description/description_en`、`base_price`
- 可配置商品：`Product.options` + 多个 `ProductSKU.option_values`
- SKU 级库存与可售状态：`stock_quantity`、`is_available`
- SKU 级多图：`ProductSKU.photos`
- 用户侧业务：注册、last address、购物车、下单、取消、确认收货
- 管理端业务：商品搜索、新增、编辑、上下架、SKU 管理、订单处理、评论治理
- 订单历史：订单项快照、收货地址快照、状态时间线
- 评论：必须关联“已购买且未取消的订单”

结论：任何单一开源“图书数据集”都只能覆盖商品元数据的一部分，无法直接满足全部业务功能。

### 2. 当前 seed 的不足

`backend/app/seed.py` 现在只生成：

- 2 个代表性图书 + 批量模板书
- 固定的平装/精装两种 SKU
- 文本占位“图片”而不是真实可展示封面
- 1 个示例订单、1 条评论

这足够演示基础流程，但不足以支撑：

- 更真实的书目检索与浏览
- 多样化作者/出版方/描述
- 大量封面与商品差异化展示
- 更接近真实业务的订单/评论/库存压力测试

## 开源数据源调研结论

### 推荐采用：混合方案，而不是单一数据源

#### 首选商品元数据源：`goodbooks-10k-extended`

来源：`https://github.com/malcolmosh/goodbooks-10k-extended`

优点：

- 约 10k 本图书，数量足够做分页、搜索、管理端测试
- 包含标题、作者、描述、语言、流行度等扩展字段
- `description` 覆盖率高，适合映射到商品详情
- 结构成熟，适合作为主要商品源

限制：

- 不包含你项目需要的双语字段，需要二次加工
- 不包含 SKU、库存、可售状态、订单、地址、评论、状态时间线
- 封面图与字段命名需要转换

#### 备选补充源：`100-best-books`

来源：`https://github.com/benoitvallon/100-best-books`

用途：

- 可作为小体量、高质量演示数据源
- 字段简单，适合验证导入逻辑或做精选样本

限制：

- 只有约 100 本书，无法单独承担长列表、分页、后台管理压力测试
- 仍然缺少 SKU / 订单 / 评论等业务数据

#### 可选封面源：`book-dataset`

来源：`https://github.com/uchidalab/book-dataset`

用途：

- 提供大规模书籍标题、作者、类别与封面 URL 线索
- 可用于后续扩充封面图来源

限制：

- 更偏研究数据集，落地接入复杂度较高
- 与当前业务字段映射成本高于 `goodbooks-10k-extended`

### 结论性推荐

最适合当前项目的方案是：

1. 用 `goodbooks-10k-extended` 作为“书籍商品元数据主来源”
2. 保留少量当前手工精选种子作为“业务覆盖样本”
3. 由本项目 seed 脚本补生成 SKU、库存、地址、订单、评论、状态流转、多图与管理员账号

若目标是“先把图片展示做真实”，则没必要先落完整数据库导入。更合适的是：

1. 保留当前 seed 与商品结构
2. 只替换/补充真实书籍封面图片
3. 继续沿用现有 SKU 本地上传路径 `/uploads/...`

这样可以以最小改动解决 demo 观感问题。

## 调整后的最小实施方案

### 方案目标

只改“图片来源与 seed 图片生成逻辑”，不优先改商品元数据来源。

该目标现在需要升级为：

- 让 seed 生成的前台商品看起来像真实书店 catalog
- 商品名称、作者、出版方、简介、封面尽量来自真实书籍资料
- 仍尽量少动现有业务模型与前端页面结构

## 调整后的总体策略

当前最合适的实现不是“纯图片替换”，而是：

1. 引入一份仓库内置的、可控的“真实书籍样本数据”
2. 每本书同时包含基础详情字段和稳定可下载/可缓存的封面来源
3. seed 优先读取本地样本数据生成商品，而不是继续批量生成 `演示图书 001`
4. 首次运行时尝试下载封面并缓存；后续运行优先使用本地缓存，弱化网络依赖

这能同时解决三个问题：

- 商品名不再重复、看起来像真实平台
- 图片不再重复、详情页不再显得空洞
- demo 不再依赖外部服务实时可用

## 外部数据源优先级（按当前用户偏好重排）

### 首选：开源数据库/公开 API 作为主来源

优先采用以下来源来获取“真实书籍信息 + 真实封面”：

1. `Open Library`
   - 优点：
     - 可通过 ISBN、OLID 等稳定拿到封面
     - 还能补到标题、作者、出版信息等基础元数据
     - 公开可访问，适合做 seed 数据来源
   - 适合用途：
     - 作为封面主来源
     - 作为书名/作者/出版社/出版信息的补充源

2. `goodbooks-10k-extended`
   - 优点：
     - 有大量真实书籍标题、作者、描述信息
     - 更适合作为商品 catalog 元数据源
   - 限制：
     - 当前环境下载稳定性一般
     - 封面字段和可直接用的图片链不如 Open Library 简洁稳定

3. `100-best-books`
   - 优点：
     - 小而精，适合精选样本
   - 限制：
     - 体量太小，不适合做主要 catalog 来源

### 次选：项目内置样本作为离线兜底

若外部源不可达，才回退到仓库内置样本 JSON 与已缓存图片。

这意味着执行时的默认顺序应是：

1. 先用开源数据库/API 获取真实书籍信息和封面
2. 再把结果固化到本地 `seed_data` 与 `seed_image_cache`
3. 后续 seed 运行优先消费这些已固化的数据

这样既满足“优先使用外部真实数据”，又兼顾 demo 稳定性。

## 推荐的数据组织方式

### 新增稳定样本文件

新增一份项目内的数据文件，例如：

- `backend/app/seed_data/books.json`

文件中每条记录至少包含：

- `title`
- `title_en`
- `author`
- `author_en`
- `publisher`
- `publisher_en`
- `description`
- `description_en`
- `category`
- `base_price`
- `isbn` 或 `cover_url`
- 可选：`cover_urls`（多张）
- 可选：`formats` / `options`

推荐首批规模：

- 30~80 本真实图书
- 先保证前 20~30 本质量高、信息全、封面稳定
- 再逐步扩展到更大数量

这样做比直接依赖远程大数据源更稳，因为：

- 你能控制哪些书出现在 demo 首页
- 你能控制每本书的中英文字段是否自然
- 你能控制详情信息是否足够像真实电商商品

补充说明：

- 该 JSON 不是“手工编造数据主来源”，而应是“从开源数据库/API 拉取后落地的稳定快照”
- 也就是：外部源优先，仓库存档次之

## 图片稳定性方案

### 从“运行时联网取图”改为“预下载缓存优先”

实现上应改为三层回退：

1. 优先使用 `seed_image_cache/` 里的本地封面
2. 本地没有时，按 `isbn` 或 `cover_url` 下载并写入 cache
3. 单本下载失败时，回退到另一张已缓存的真实书籍封面，而不是 demo 图或 txt 占位

重点：

- 前端永远读取 `/uploads/...`
- seed 只负责把缓存图复制到 SKU 目录
- 真正的不稳定环节只发生在“首次缓存下载”阶段

用户当前偏好下，这里的更准确表述应为：

- 首次构建数据时，优先从外部开源源获取图片和信息
- 一旦获取成功，就把它们缓存/固化到项目中，避免之后每次 seed 都重新依赖网络

### 不推荐继续使用单一 cover pool 轮转

当前 `COVER_POOL` 只有少量封面 URL，且商品名称还是大量模板书，这会导致：

- 同一批图片在几十本商品中反复轮转
- 商品名和封面语义不匹配
- 一眼看上去像 mock 数据

因此执行阶段应废弃“少量图片池循环复用”的思路，改成“书籍记录和封面一一绑定”。

## 商品真实性方案

### 真实书名 + 真实 detail 信息

要让项目看起来像正常购书平台，seed 不能只换图，必须同步替换以下字段：

- `Product.title` / `title_en`
- `author` / `author_en`
- `publisher` / `publisher_en`
- `description` / `description_en`
- 必要时 `category_id`

原因：

- 详情页会显示作者、出版社、描述
- 后台商品管理也直接展示这些字段
- 如果图片真实、但文案还是“演示图书 023 / Demo Author”，违和感依然很强

因此执行上应避免“只从外部拿图，不从外部拿信息”的半成品方案。
更合理的是：

- 书名、作者、出版社、简介、封面尽量来自同一真实来源链
- 至少保证前 20~30 本商品的数据是一致且可信的

### 双语字段现实化策略

推荐采用“英文原始信息 + 中文精选翻译”的混合做法：

- 对首批重点展示书籍，人工提供高质量中文名、中文作者名、中文出版社和中文简介
- 其余书籍允许中文字段暂时等于英文字段，保证 schema 可通过

推荐最低标准：

- 前 20~30 本图书具备较自然的中英双语信息
- 剩余书籍至少具备真实英文书名、作者、简介和封面

这样切换语言时，前几页会更像真实平台。

## SKU 与图片绑定策略

### 按书籍记录生成 SKU，而不是固定“平装/精装+随机图”

可继续保留现有 `平装/精装` SKU 业务模型，但图片应按书绑定：

- 同一本书的两个 SKU 共享同一组封面主图是可以接受的
- 若样本数据里有多张图，可让精装/平装各自拥有不同图组
- 若只有 1 张真实封面，可通过裁剪/重复同图作为保底，但只对少量商品这样做

更自然的做法：

- 对大多数书：1~2 张图
- 对重点展示书：2~4 张图
- 对少量商品维持多 SKU 差异化图组，用于演示 SKU 切图

## 推荐实施顺序（已根据新诉求重排）

1. 先确定外部主数据源组合：`Open Library` 负责封面与基础书目信息，`goodbooks-10k-extended` 可选补充描述字段
2. 编写一次性的“拉取与固化”流程，把外部真实书籍数据落成 `seed_data/books.json`
3. 为每本书记录稳定标识（如 `isbn` / `olid` / `cover_url`）用于后续复用
4. 重构 `seed.py`，从落地后的数据文件生成 Product，而不是模板字符串生成
5. 重构封面缓存逻辑：按书缓存，不按共享图片池缓存
6. 重构 `create_product_with_skus()`：让商品字段来自样本记录
7. 保留现有订单/评论/地址/管理端演示逻辑，但让它们关联到真实书籍商品
8. 验证首页前 2~3 页商品是否都像真实书店 catalog

## 具体执行建议

### 样本数据来源建议

由于当前环境联网并不稳定，执行阶段应优先采用“仓库内固定样本 + 可选下载增强”模式：

- 先通过开源数据库/API 抓取一批真实书籍数据
- 再把结果固化成仓库中的 `books.json`
- 运行 seed 时优先使用这份已固化的真实元数据
- 对封面：
  - 若 cache 已存在，直接用本地文件
  - 若 cache 不存在，再按记录中的外部来源重新下载

这样就算离线，也能维持较好的 demo 效果。

### 数据文件内容质量要求

为避免“像临时拼凑的 mock 商店”，样本数据最好覆盖：

- 文学、科幻、历史、商业、技术、儿童等多个类别
- 中外经典和当代畅销书混合
- 作者、出版社、简介都像真实书籍 catalog 文案
- 价格有自然分布，不要全是机械递增

### 图片命中率要求

执行时应把“真实封面命中率”作为硬指标：

- 前 30 本商品：目标 100% 真实封面
- 全量默认 seed 商品：至少 80% 使用与书籍匹配的真实封面
- 剩余未命中的，也必须回退到另一张真实书籍封面，而不是纯 demo 图

## 验收标准（升级版）

最终结果应满足：

- 商品列表前几页不再出现 `演示图书 001`、`Demo Book 001` 这类明显 mock 文案
- 商品封面与书名、详情内容基本匹配，不再是少量图片反复复用
- 商品详情页展示真实作者、出版社、简介，视觉和信息层面都像真实购书平台
- 后台商品列表中也能看到真实书名与作者，而不是演示字符串
- 重新运行 seed 后，图片与商品的对应关系稳定，不会随机漂移
- 即使网络不稳定，只要已有 cache，demo 仍能完整展示

## 关键实现决策

若进入执行阶段，默认按以下决策落地：

- 不再以 `演示图书 NNN` 作为主商品来源
- 引入“由开源数据库/API 生成并固化”的真实书籍样本 JSON 文件
- 以“书籍记录绑定封面”的方式替代“共享 cover pool 轮转”
- 以开源外部源为主来源、以本地 cache 为稳定性保障
- 前 20~30 本作为高质量 demo 样本，优先保证真实度

### 推荐图片来源

优先顺序建议如下：

1. 使用可公开访问的书籍封面 URL 批量下载为本地图片
2. 下载后缓存到 `backend/app/uploads/sku_{id}/...`
3. seed 将 `sku.photos` 指向这些真实 jpg/png/webp 文件

推荐可用来源思路：

- `Open Library Covers API`：适合通过 ISBN / OCLC / Open Library ID 获取封面
- `goodbooks-10k-extended` 或其他公开书单中的封面字段/关联字段：作为候选元数据入口
- 仓库中少量手工维护的真实书籍封面：作为稳定 demo 样本

关键原则：

- 前端最终只读本地 `/uploads/...`
- demo 不依赖运行时外链图片
- 即使网络断开，demo 图片也能正常显示

### 为什么不直接前端用远程图链

当前项目后端与前端已经围绕本地 `/uploads/...` 建立了完整图片机制：

- 商品列表缩略图从 SKU photos 推导
- 商品详情页按 SKU 展示图片组
- 管理端支持上传/删除图片

若直接改成远程 URL：

- 会破坏现有“本地上传/删除”的业务语义
- demo 时容易因外链失效、跨域或限流导致翻车
- 无法体现后台图片管理功能已经完成

因此最稳妥的方案仍然是“预下载到本地，再通过现有 uploads 机制展示”。

## 针对当前需求的具体改造清单

### 阶段 1：准备真实图片样本

准备一批真实书籍封面图，建议：

- 至少 20~30 本书有真实封面
- 其中 5~10 本书有 2 张以上图片，覆盖轮播展示
- 与当前 seed 中的精选商品做一一对应，保证 demo 路径稳定

推荐最低 demo 标准：

- 首页/商品列表看到的前 20 本都有真实封面
- 详情页至少若干商品具备多图
- 管理端打开 SKU 图片管理时，能看到真实图并可删除

### 阶段 2：替换 seed 中的伪图片写法

当前 `backend/app/seed.py` 用 `_write_dummy_image()` 写 `.txt` 文件，这会导致浏览器不能当图片显示。

执行时应改为：

- 从本地图片素材池复制真实图片到 `backend/app/uploads/sku_{id}/`
- 或从公开来源下载真实封面到该目录
- `sku.photos` 写入真实图片路径列表，如 `/uploads/sku_12/cover_1.jpg`

核心结果：

- 不再生成 `.txt` 文件作为图片
- 所有种子商品都至少有 1 张合法图片文件

### 阶段 3：建立“稳定 demo 图库”而不是完全随机图源

为避免 demo 失控，建议把图片分成两层：

- 固定样本层：10~30 本固定图书，图片人工确认可用
- 扩展层：其余商品可复用固定图库或按规则分配图片

这样可以保证：

- 你演示时最先看到的商品一定有真实图片
- 不需要一次性为所有 200 本都找唯一封面
- 图片功能观感已经足够完成态

### 阶段 4：多 SKU 图片策略

当前代码支持 SKU 级多图，demo 时应刻意保留几类样本：

- 单 SKU + 1 张图
- 单 SKU + 多张图
- 多 SKU + 每个 SKU 不同图
- 多 SKU + 每个 SKU 多张图

这样可完整展示：

- 列表缩略图
- 商品详情轮播
- 切换配置时图片跟着变化
- 后台对某个 SKU 的图片管理

## 图片来源的实施建议

### 最稳妥方案

采用“少量人工筛选真实封面 + 本地缓存”的方案。

理由：

- 最适合课程 demo
- 图片质量可控
- 不受第三方服务波动影响
- 实施量小，见效最快

### 次优方案

写一个下载脚本，从公开来源抓取封面后缓存到本地，再由 seed 读取。

适合：

- 你希望商品数量较多
- 你接受部分图下载失败时需要回退处理

### 当前阶段不推荐

- 继续使用 `.txt` 伪图片
- 继续使用纯色/文字占位图作为主要展示资源
- 前端直接依赖外链图片而不落地本地缓存

## 调整后的验收标准

完成后至少应满足：

- `frontend/src/views/Products.vue` 中商品列表展示真实书籍封面
- `frontend/src/views/ProductDetail.vue` 中商品详情展示真实图片，而不是文本占位
- `frontend/src/components/ProductImageGallery.vue` / 轮播区域能正常切换真实图片
- `frontend/src/views/admin/AdminProducts.vue` 的 SKU 图片管理能看到真实封面
- 运行 `python -m app.seed` 后，`backend/app/uploads/` 下生成的都是合法图片文件而不是 `.txt`
- demo 首页前几屏商品都具备真实图片，视觉上不再像“功能未完成”

## 实施方案

### 阶段 1：定义最终导入数据模型

先明确开源字段如何映射到现有模型：

- `title` -> `Product.title_en` 或 `Product.title`（取决于源数据语言）
- `authors` -> `author_en` / `author`
- `description` -> `description_en` / `description`
- `publisher` / `publishDate` / `genres` -> 视项目字段与 UI 需要择优映射
- 中文字段缺失时：
  - 方案 A：用英文自动回填中文字段，保证 schema 可通过
  - 方案 B：对精选样本做人工/规则翻译，其余英文回填

推荐默认：

- 全量数据采用“英文字段为主，中文字段先回填英文”
- 额外保留 20~50 本人工精选双语样本，专门验证 W2 展示逻辑

原因：

- 当前后端 schema 强制管理端商品需双语字段
- 若追求全量高质量中英双语，成本远高于项目收益
- 用“精选双语样本 + 全量英文回填”更符合课程项目投入产出比

### 阶段 2：重构 seed 输入来源

将 `backend/app/seed.py` 从“纯代码内联模板生成”改为“数据驱动 + 业务补全”：

- 新增一个原始数据目录，例如 `backend/app/seed_data/`
- 存放：
  - 开源图书 CSV 的裁剪版或预处理版
  - 少量本地封面图/占位图
  - 精选双语修正映射表
- seed 脚本读取这些文件，再写入 DB

注意：

- 不建议把 10k 原始完整 CSV 直接完整提交进仓库
- 更适合提交“裁剪后的测试集”，例如 200~1000 本
- 若担心仓库体积，可提供下载脚本 + 本地缓存机制

推荐规模：

- 默认 seed：200 本商品
- 可通过环境变量扩展到 1000 本
- 保留当前 `SEED_PRODUCT_COUNT` 风格配置

### 阶段 3：补齐商品业务字段

对每本导入图书自动补全当前业务所需字段：

- `base_price`：按规则生成，如基于页数/评分区间/随机种子稳定生成
- `options`：并非所有图书都必须复杂配置，但为了覆盖 D1-D5，建议混合生成：
  - 70%：单 SKU 简单商品（`options` 为空或 `{}`）
  - 30%：双 SKU 或三 SKU 商品，例如 `edition` / `format`
- `option_values`：如 `{"format":"paperback"}`、`{"format":"hardcover"}`
- `price_adjustment`：精装、礼盒等版本高于基础价
- `stock_quantity`：生成 0、低库存、正常库存三类样本
- `is_available`：保留一部分 false，用于前台/后台缺货与禁售测试

关键点：

- 必须有部分商品是无库存或不可售，才能覆盖 D5
- 必须有部分商品是多 SKU，才能覆盖 D1-D4
- 必须有足够多商品数量，才能覆盖 A5 分页与 A15 后台搜索

### 阶段 4：处理图片策略

当前项目的图片逻辑是 SKU 级本地文件路径 `/uploads/...`，不是外链字段。

因此建议不要直接依赖远程图片 URL，而采用两层策略：

- 默认方案：seed 下载或复制本地封面图到 `backend/app/uploads/sku_{id}/...`
- 回退方案：若外部图片不可用，则写入本地统一占位图而不是 `.txt` 文件

推荐实现：

- 预置 10~30 张合法 jpg/png 占位封面图
- 对真实开源封面可选下载；下载失败则回退占位图
- 每个 SKU 至少 1 张图，多 SKU 商品为不同版本分配不同图；少量 SKU 分配 2~3 张图以覆盖轮播

这样可以真正测试：

- 列表缩略图
- 详情轮播
- SKU 切换图片
- 管理端图片删除/展示

### 阶段 5：补齐用户、地址、购物车、订单、评论数据

开源书籍数据集无法提供这些业务数据，必须由本项目 seed 合成。

建议生成以下测试数据层级：

- 账号：
  - 1 个管理员
  - 3~10 个普通用户
- 地址：
  - 每个普通用户至少 1 条 last address
  - 部分用户保留历史地址，验证兼容逻辑
- 购物车：
  - 给 1~2 个用户预置购物车项，覆盖 A8-A10
- 订单：
  - `pending`
  - `shipped`
  - `completed`
  - `cancelled`
  - 超过 3 天待自动取消的 `pending`
- 评论：
  - 仅为已购买且未取消订单生成评论
  - 保留可隐藏/可见两类，覆盖管理端评论治理

订单数据需满足：

- 有地址快照
- 有订单项配置快照
- 有状态时间线 `OrderStatusEvent`
- 有库存扣减与取消返还后的合理结果

### 阶段 6：国际化数据策略

这是当前项目最容易被忽略，但必须满足的部分。

项目要求商品双语字段，而外部数据大多只有英文，因此建议：

- 方案主线：英文源数据写入 `*_en`
- 中文字段先按规则回填英文，保证系统可运行
- 对首页热门书、精选书、后台演示书做人工中文映射，确保：
  - 商品列表语言切换明显可见
  - 商品详情中作者/出版社/描述切换明显可见
  - 评审时可直接展示 W2 效果

推荐最小可行标准：

- 至少 30 本书具备真实中英双语
- 其余书籍允许中文字段临时等于英文字段

这样可以满足“全部功能可运行”与“国际化验收可展示”。

### 阶段 7：保留现有业务样本，避免纯随机数据破坏验收可控性

建议不要完全替换当前 seed，而是分层：

- 第一层：固定演示样本
  - 管理员账号
  - 普通用户账号
  - 几本精心准备的双语、多 SKU、多图书籍
  - 几条确定状态的订单与评论
- 第二层：批量开源图书导入
  - 用于分页、搜索、压力测试、后台管理列表

这样做的价值：

- 手动验收路径稳定
- 问题排查容易
- 不会因上游数据质量波动导致演示失败

## 风险与规避

### 1. 数据许可证风险

实施前必须确认最终采用数据源的 license 或使用说明，尤其是：

- 是否允许再分发到课程仓库
- 是否允许下载和缓存封面图

规避策略：

- 元数据可提交，图片尽量用本地占位图或用户自行下载脚本
- 在 README 中注明来源与 license

### 2. 图片失效风险

远程封面链接可能失效。

规避策略：

- seed 以本地缓存为准
- 下载失败时优先复用另一张真实书籍封面，而不是文本占位文件

### 3. 双语质量风险

全量自动翻译成本高，也可能引入不稳定性。

规避策略：

- 精选样本人工双语
- 大盘数据英文回填中文字段

### 4. 数据量过大导致 seed 太慢

规避策略：

- 默认导入 200 本
- 通过环境变量扩容
- 将预处理结果缓存为轻量文件

## 最终建议

建议按以下方案落地：

- 当前阶段主任务不是“换数据库”，而是“让图片看起来真实完成”
- 保留现有商品与业务 seed，不先大改数据库结构
- 图片策略改为：真实书籍封面本地缓存到 `backend/app/uploads/`
- 默认优先保证首页、详情页、后台图片管理这三条 demo 路径效果

这不是“直接找一个现成数据库导入就完事”的项目。对当前代码库来说，正确做法是“开源图书元数据 + 项目内业务种子生成器”的混合数据库方案。只有这样，才能真正满足现在的业务逻辑和全部功能。

- 优先替换 `backend/app/seed.py` 中的 `.txt` 伪图片逻辑
- 使用真实书籍封面并缓存到本地 `uploads`
